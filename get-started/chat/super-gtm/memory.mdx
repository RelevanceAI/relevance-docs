Memory is how the agent learns about you across conversations. It stores your preferences, corrections, project context, and references so the agent gets better over time — without you having to repeat yourself.

User memory is loaded at the start of every conversation. The agent always has your context before it starts working.

## Where to find memory

<Frame>
  <img src="/images/where-to-find-memory.png" alt="Context and Memories section in the Overview tab" />
</Frame>

You can view and manage your memories in the **Overview** tab, under **Context & Memories**. From here you can see all stored memories, review what the agent has learned about you, and remove anything that's outdated or incorrect.

<Note>
Memory is scoped per user. Your memory is not shared with other users in your project.
</Note>

## How it works

The agent maintains a memory entry for each thing it learns about you. At the start of each conversation, a lightweight model scans your memories and picks the ones relevant to what you're asking about. Those are loaded into the agent's context alongside your message.

The agent doesn't load everything every time — it loads what's relevant. The more clearly your memories are described, the better the selection works.

## Memory types

<AccordionGroup>
  <Accordion title="User memory">
    Information about you — your role, goals, expertise, and preferences. The agent uses this to tailor its work to who you are.

    **Examples:**
    - "VP of Sales at Relevance AI, manages a team of 8 AEs focused on enterprise accounts"
    - "Prefers data-driven insights and concise communication"
    - "Has deep HubSpot experience but is new to using Slack integrations"
  </Accordion>
  <Accordion title="Feedback memory">
    Corrections and guidance you've given the agent. These are the most impactful memories — they prevent the agent from repeating mistakes.

    **Examples:**
    - "Keep email drafts professional but warm — no corporate jargon. Start with the contact's first name, not 'Dear [Name]'"
    - "Don't include pricing in the first outreach email"
    - "Always present data as bullet points, not paragraphs"
  </Accordion>
  <Accordion title="Project memory">
    Context about ongoing work, goals, and timelines that the agent wouldn't otherwise know.

    **Examples:**
    - "Enterprise team has a $2.4M pipeline target for Q3, closing September 30. Board review is October 5"
    - "The Q2 campaign is focused on enterprise expansion — prioritize enterprise accounts in any outreach"
    - "Product launch is scheduled for June 15 — all deliverables need to be finalized by June 10"
  </Accordion>
  <Accordion title="Reference memory">
    Pointers to where information lives, so the agent knows where to look.

    **Examples:**
    - "Deal stage definitions are in the HubSpot wiki under 'Sales Process > Deal Stages'"
    - "The competitive analysis is saved in project files under /research/competitors.md"
    - "Customer feedback is tracked in the Notion database called 'Product Feedback'"
  </Accordion>
</AccordionGroup>

## Building up memory over time

You don't need to set up memory all at once. The agent builds it naturally as you work together.

<Tabs>
  <Tab title="Tell it to remember">
    Ask the agent to save something to memory at any time:

    > "Remember that I prefer bullet points over paragraphs"

    > "Remember that the Q3 target is $2.4M"

    The agent will confirm what it stored and which memory type it used.
  </Tab>
  <Tab title="Correct it">
    When you correct the agent — "No, don't include pricing in the first email" — it saves a feedback memory so it doesn't make the same mistake next time.
  </Tab>
  <Tab title="Tell it to forget">
    If a memory is outdated or incorrect:

    > "Forget what you know about the Q2 campaign"

    The agent will find and remove the relevant memory.
  </Tab>
</Tabs>

## How skills and memory work together

[Skills](/get-started/chat/super-gtm/skills) and memory complement each other:

- **Skills define the process.** A skill tells the agent *how* to do something — the steps, the data sources, the output format.
- **Memory personalizes the execution.** Memory tells the agent *your way* of doing it — your tone, your preferences, your priorities.

A well-written skill references memory where it matters. For example, a post-meeting follow-up skill tells the agent to check your memory for communication preferences before drafting emails. The skill is the same for everyone on the team, but the output adapts to each person's style.

This means you can share a project skill with your whole team, and each person's agent executes it differently based on their individual memory.
